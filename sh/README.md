

# Library Dependency Analyzer

Скрипт для анализа зависимостей бинарных файлов в Linux (Debian/Ubuntu). Определяет, какие shared libraries использует бинарный файл, и находит соответствующие deb-пакеты.

## Возможности

- 🔍 Анализ библиотек с помощью `ldd`
- 📦 Определение deb-пакетов для каждой библиотеки
- 🔗 Корректная обработка символических ссылок
- 🎯 Формирование уникального списка пакетов (без дубликатов)
- 📄 Подробное документирование всех шагов
- 📊 Вывод статистики по зависимостям
- 🖥️ Специальная обработка Qt-библиотек

## Требования

- **ОС**: Debian/Ubuntu или любая система с dpkg
- **Права**: Рекомендуется запуск с `sudo` для полного доступа к базе данных пакетов
- **Утилиты**: 
  - `ldd` (обычно входит в пакет libc-bin)
  - `dpkg` (системная утилита)
  - `readlink` (coreutils)
  - `apt-file` (опционально, для расширенного поиска)

Как это работает

    Анализ библиотек: Использует ldd для определения всех shared libraries, необходимых бинарному файлу

    Извлечение путей: Парсит вывод ldd и извлекает полные пути к библиотекам

    Разрешение символических ссылок: С помощью readlink -f преобразует символические ссылки в реальные пути

    Поиск пакетов: Использует dpkg -S для определения, какому пакету принадлежит каждый файл

    Дедупликация: Сортирует и удаляет дубликаты пакетов

    Документирование: Сохраняет все шаги в выходной файл

Особенности обработки
Символические ссылки

Скрипт корректно обрабатывает символические ссылки, например:

    /lib/x86_64-linux-gnu/libm.so.6 → /usr/lib/x86_64-linux-gnu/libm.so.6

    Пакет libc6 будет найден для обоих путей

Специальные случаи

Для часто встречающихся библиотек добавлена ручная обработка:

    libc.so.*, libm.so.*, libresolv.so.* → libc6

    libgcc_s.so.* → libgcc-s1

    libcrypto.so.* → libssl3

    libstdc++.so.* → libstdc++6

    libz.so.* → zlib1g

Возможные проблемы и решения
Проблема: "dpkg-query: не найден путь"

Причина: Недостаточно прав или файл является символической ссылкой
Решение: Запустите скрипт с sudo
Проблема: Некоторые пакеты не найдены

Причина: Библиотеки из системных пакетов (libc6, libgcc-s1 и т.д.)
Решение: Скрипт автоматически обрабатывает эти случаи в специальных правилах
Проблема: "ldd: not found"

Причина: Отсутствует утилита ldd
Решение: Установите пакет libc-bin:
bash

sudo apt install libc-bin

Проблема: Медленная работа

Причина: Поиск по всей базе dpkg
Решение: Используйте apt-file для ускорения:
bash

sudo apt install apt-file
sudo apt-file update

Практические примеры использования
1. Установка всех зависимостей приложения
bash

# Анализ приложения
sudo ./analyze_libs.sh ./myapp

# Установка всех найденных пакетов
sudo apt install $(cat analysis_myapp_*_packages.txt)

2. Проверка установленных пакетов
bash

# Создание списка пакетов
sudo ./analyze_libs.sh /usr/bin/python3

# Проверка установленных пакетов
dpkg -l | grep -Ff analysis_python3_*_packages.txt

3. Создание списка для другой системы
bash

# Сохраняем список пакетов
sudo ./analyze_libs.sh ./app ./dependencies.txt

# На другой системе устанавливаем
sudo apt install $(cat dependencies.txt)

4. Анализ всех бинарных файлов в директории
bash

for file in /usr/local/bin/*; do
    sudo ./analyze_libs.sh "$file"
done
